# Student Election Portal (Student Voting System)

A PHP + MySQL web application that lets students **register, log in, vote for a candidate/group, and view live election results**, with a separate **Admin panel** to approve or reject candidates. Built to run on **XAMPP** (Apache + MySQL + PHP).

## Features

### Voter
- Register with name, mobile number, password, and a profile photo (`api/register.php`).
- Log in with mobile number + password, selecting role (**Voter** or **Candidate**) from a dropdown (`api/login.php`).
- View a personal dashboard showing profile photo, name, mobile number, and voting status (**Voted** / **Not voted**) (`routes/dashboard.php`).
- View all candidate groups and cast a single vote. Once voted, the **Vote** button is replaced with a disabled **Voted** button.
- View election results after a configured results date (`routes/result.php`).
- Log out (`routes/logout.php`).

### Candidate
- Registers through the same registration form, selecting the "Candidate" role.
- Appears in the voter dashboard as a group/candidate to vote for, once approved by the admin.

### Admin
- Separate hardcoded admin login (`admin/login.php` → `admin/verify.php`).
- View and **approve/reject** pending candidate registrations (`admin/dashboard.php`, `api/adminActions.php`).
- View list of approved candidates with their current vote counts.
- Log out.

### Results
- `routes/result.php` displays results only after a hardcoded date (`$result_date = "2025-04-29"`); before that, it shows a "coming soon" message.
- Candidates (role = 2) are ranked by votes; the top candidate is highlighted with a gold "winner" row and a 🏆 trophy icon.

## Setup Instructions (XAMPP)

1. **Install XAMPP** (if not already installed) and start the **Apache** and **MySQL** modules from the XAMPP Control Panel.

2. **Copy the project folder** into your XAMPP `htdocs` directory, e.g.:
   ```
   C:\xampp\htdocs\voting\
   ```
   or on Linux/Mac:
   ```
   /opt/lampp/htdocs/voting/
   ```

3. **Create the database:**
   - Open `http://localhost/phpmyadmin`.
   - Create a new database named **`voting`** (this name is hardcoded in `api/connect.php`).
   - Create the `user` table using the schema above (or import your own SQL dump if you have one).

4. **Check database credentials** in `api/connect.php`:
   ```php
   $connect = mysqli_connect("localhost", "root", "", "voting")
   ```
   Default XAMPP credentials are `root` with no password — update this if yours differ.

5. **Create the uploads folder** (if not already present) and ensure it's writable, since `api/register.php` uses `move_uploaded_file()` to store profile photos there:
   ```
   /uploads/
   ```

6. **Set the results date** (optional): open `routes/result.php` and adjust:
   ```php
   $result_date = "2025-04-29";
   ```
   Results will only display on/after this date.

7. **Access the app** in your browser:
   ```
   http://localhost/voting/index.html
   ```

8. **Admin panel** is accessed separately at:
   ```
   http://localhost/voting/admin/login.php
   ```
   using the hardcoded credentials above.

## Usage Flow

1. A new user registers via the registration form, choosing role **Voter** or **Candidate**, and uploads a profile photo.
2. If registering as a **Candidate**, an admin must approve them via the Admin Dashboard before they appear as a votable group.
3. A **Voter** logs in, sees their dashboard with all approved candidates, and casts one vote.
4. Once voted, their status changes to "Voted" and the vote button is disabled/replaced.
5. After the configured results date, anyone can view the results page showing rankings and the winning candidate.

## Security Notes (for future improvement)

These are observations from the code, included for awareness — not implemented changes:

- **Passwords are stored and compared in plain text** (no hashing, e.g. `password_hash()`/`password_verify()`).
- **Admin credentials are hardcoded** in source code rather than stored securely or hashed.
- User input for login/registration is escaped with `mysqli_real_escape_string()`, which reduces but does not fully eliminate SQL injection risk compared to prepared statements.
- Uploaded photo filenames are used as-is (`$_FILES['photo']['name']`) without sanitization or randomization, which could allow filename collisions or path traversal if not further restricted.
- `api/vote.php` accepts `gvotes` (current vote count) from a hidden form field rather than re-fetching it server-side, which could be manipulated by a user tampering with the POST request.
